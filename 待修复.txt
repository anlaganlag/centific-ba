BA Toolkit 项目评估报告

  评估视角：内部工具 + 业务交付价值优先

  ---
  一、总体评价

  这是一个完成度较高的内部工具原型，核心业务流程完整可用。 3 步 AI 分析流水线（特征提取 → 访谈问答 →
  用户故事生成）设计合理，auto/guided 双模式满足了不同场景的需求。作为内部工具，不需要按面向公众产品的标准苛求，但有几个
  问题会直接影响业务交付的可靠性和效率。

  评级：B- — 能用，但存在影响业务信任度的风险点。

  ---
  二、从业务交付价值角度看核心问题

  问题 1：分析结果不可追溯、不可对比（影响：高）

  现状：
  每个项目只返回最新一次分析结果（get_latest_analysis_session，db_service.py:233）。重新运行会覆盖上一次的可见结果。

  业务影响：
  - BA 无法对比两次分析的差异（如文档更新后重跑）
  - 没有版本历史，无法回答"上次分析为什么不同"
  - Guided 模式下用户花时间填写的答案，重跑后丢失访问入口

  建议： 前端增加 session 列表和切换，成本很低但价值显著。

  ---
  问题 2：AI 输出质量没有护栏（影响：高）

  现状： 完全依赖 LLM 的结构化输出，没有业务层面的校验。

  - feature_extraction_agent.py:163 — reduce agent 返回什么就存什么，不校验 feature_id 格式是否合规、是否有重复
  - story_generation_agent.py:105 — asyncio.gather 不处理单个 feature 失败的情况（一个报错全部失败）
  - Given/When/Then 格式没有验证，可能输出自由文本

  业务影响：
  - 导出的 DOCX 可能包含格式混乱的内容，交付给客户时损害专业形象
  - 一个 chunk 的 LLM 调用失败会导致整个分析失败，但可能 90% 的内容已经成功提取了

  建议： 至少加 return_exceptions=True 后的容错处理（story_generation_agent.py 第 105 行已经缺失这个），以及对输出的
  feature_id/story_id 做格式校验。

  ---
  问题 3：长文档场景下的 Token 经济性问题（影响：中高）

  现状：
  - 分块大小 3000 字符固定（document_service.py），按字符切割不按语义
  - 每个 chunk 独立调用 GPT-4o map agent（feature_extraction_agent.py:127）
  - 一份 50 页 PDF 可能产生 50+ chunks → 50+ 次 GPT-4o 调用
  - 面试和故事生成阶段的文档上下文被截断到 15,000 字符（analysis_service.py:22）

  业务影响：
  - 大文档分析一次的 API 费用可能达 $5-15（GPT-4o 按 token 计费）
  - 用户无法估算成本，也没有用量统计
  - 截断 15K 字符意味着超过 ~20 页的文档内容在后续步骤中被丢弃，影响故事质量

  建议： 加一个分析前的预估（chunk 数 × 预估 token），以及对大文档的预警机制。

  ---
  问题 4：错误恢复能力差（影响：中）

  现状：
  - analysis_service.py:165-171：任何异常直接标记整个 session 为 error，没有部分结果保留
  - asyncio.create_task 启动后台任务（第 77 行），如果进程重启，正在进行的分析无法恢复
  - 前端轮询发现 error 后只显示 str(e)，用户不知道如何处理

  业务影响：
  - Azure OpenAI 偶发 429/500 错误时，5 分钟的分析全部白费
  - BA 只能重跑，但无法知道是"文档有问题"还是"API 暂时不可用"

  ---
  问题 5：文档删除时磁盘文件泄漏（影响：低但应修）

  documents.py 路由中删除文档只删 DB 记录和 ChromaDB 向量，不删磁盘上的实际文件。长期运行后 data/uploads/ 会无限增长。

  四、作为内部工具可以接受的"非最佳实践"

  以下通常被视为问题，但对于内部工具来说是合理的权衡：

  - SQLite 而非 PostgreSQL — 内部少量用户场景下完全够用
  - 无数据库迁移工具（Alembic） — CREATE TABLE IF NOT EXISTS + try/except ALTER TABLE 对内部工具可接受
  - localStorage 存 JWT — 内部网络环境下风险可控
  - 无 rate limiting — 内部用户可控
  - 无国际化 — 目标用户明确
  - 轮询而非 WebSocket — 2 秒间隔的轮询对几个并发用户完全可以
  - 最小化测试覆盖 — 有 21 个基本测试覆盖核心路径，对内部工具够用

  ---
  六、结论

  这个项目的业务设计思路是对的 — 3 步流水线精准命中了 BA 工作中最耗时的部分（从文档中提取特征 → 生成问题 →
  写用户故事）。代码质量中上，架构清晰。

  最大的风险不在技术债务，而在业务信任度：如果分析偶发失败且无法恢复、如果输出质量不稳定、如果历史结果无法追溯，BA
  会回退到手工方式——此时工具的业务价值归零。

  建议优先解决 P0 的两个容错问题，这是让工具从"能跑的 demo"变成"可依赖的工具"的关键一步。
